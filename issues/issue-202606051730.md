あなたはエキスパートのフルスタックエンジニアです。
先ほど作成したMisskey PWAクライアントプロジェクトについて、デプロイ先を Cloudflare Pages から **Cloudflare Workers (Static Assets対応)** へ移行したいと思います。
今後のCloudflareのプラットフォーム統合の流れや、CORS回避のためのAPIプロキシ実装の容易さを考慮した方針転換です。

現在のプロジェクト構成（React + Vite + Hono）を維持したまま、ステップバイステップで以下の要件を満たすようにコードと設定を修正してください。

# 変更・追加の要件
1. **Cloudflare Workers (Static Assets) への移行**:
   - `wrangler.toml` をPages構成からWorkers構成に書き換える。`assets = { directory = "./dist/client" }` のような最新の静的アセット配信設定を利用する。
   - ViteとHonoのビルド設定（`vite.config.ts`）をWorkers向けに変更する（`@hono/vite-build/cloudflare-pages` ではなく、Workers向けの設定または `hono/cloudflare-workers` を使用する構成へ変更）。
2. **APIプロキシ（CORS回避）の実装**:
   - Honoのサーバーサイド（`src/server/index.ts`）に、Misskey APIへのリクエストを中継するリバースプロキシエンドポイント（例: `/api/proxy/*`）を実装する。
3. **フロントエンドの通信先変更**:
   - `misskey-js` のAPIクライアント（`src/client/lib/misskey.ts`）が、直接Misskeyインスタンスを叩くのではなく、Honoで作成したプロキシを経由して通信するように設定を変更する。
4. **GitHub Actionsの設定**:
   - Cloudflare Workersへの自動デプロイを行うためのGitHub Actionsワークフローファイル（`.github/workflows/deploy.yml`）を新規作成する。

# 実装ステップ
以下の順番で既存コードの修正と追加を行ってください。各ステップが終わるごとにコードを提示し、次のステップに進んでください。

- [Step 1] 設定ファイルの修正: `wrangler.toml` と `vite.config.ts` を修正し、Workers (Static Assets) としてビルド・デプロイできる構成にする。
- [Step 2] バックエンドの拡張: Hono (`src/server/index.ts`) にMisskey APIへのリバースプロキシ機能を実装する。
- [Step 3] フロントエンドの修正: `src/client/lib/misskey.ts` を修正し、API通信やWebSocketがHonoのプロキシを経由するように変更する。
- [Step 4] CI/CDの設定: GitHub Actionsを利用したデプロイ用ワークフローファイル (`.github/workflows/deploy.yml`) を作成する。
